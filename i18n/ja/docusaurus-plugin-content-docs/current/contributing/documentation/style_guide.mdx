# LangChain ドキュメンテーションスタイルガイド

## はじめに

LangChainが成長するにつれ、それをカバーするためのドキュメンテーションの範囲も広がり続けています。
このページでは、LangChainのドキュメンテーションを書く際のガイドラインと、組織化と構造に関する私たちの哲学を提供しています。

## 哲学

LangChainのドキュメンテーションは、[Diataxis framework](https://diataxis.fr)に従うことを目指しています。
このフレームワークの下では、すべてのドキュメンテーションは以下の4つのカテゴリーのいずれかに分類されます:

- **チュートリアル**: 概念的なステップを踏んでプロジェクトを完成させるよう読者の手を取るレッスン。
  - 例としては、[LCEL ストリーミングガイド](/docs/expression_language/streaming)があります。
  - [カスタムコンポーネント](/docs/modules/model_io/chat/custom_chat_model)に関するガイドもこれに該当します。
- **ハウツーガイド**: 現実世界の問題を解決するための手順を示すガイド。
  - 最も明確な例は、[ユースケース](/docs/use_cases/)クイックスタートページです。
- **リファレンス**: 仕組みの技術的な説明と操作方法。
  - [Runnable インターフェース](/docs/expression_language/interface)ページがその例です。
  - [APIリファレンスページ](https://api.python.langchain.com/)もこれに該当します。
- **説明**: 特定のトピックを明確化し、深く理解させるための説明。
  - [LCELプリミティブページ](/docs/expression_language/primitives/sequence)がその例です。

各カテゴリーは固有の目的を持ち、コンテンツの書き方と構造化に特定のアプローチが必要です。

## 分類

上記を念頭に置いて、LangChainのドキュメンテーションを分類しました。新しいドキュメンテーションを作成する際は、これらの観点を意識すると良いでしょう:

### はじめに

[はじめに](/docs/get_started/introduction)セクションには、LangChainの概要、各機能をツアーするクイックスタート、インストールやプロジェクト設定に関する手順などが含まれています。

ここには**ハウツーガイド**と**説明**の要素が含まれています。

### ユースケース

[ユースケース](/docs/use_cases/)は、LangChainを使ってある特定のタスク(RAG、情報抽出など)を実行する方法を示すガイドです。
クイックスタートは、実践的なプロトタイプを作成し、その後に各パーツを振り返って学習したい初心者LangChain開発者にとって良い入り口となるはずです。これらは、LangChainが得意とするものを反映するべきです。

ここのクイックスタートページは**ハウツーガイド**に該当し、その他のページは、メインのハッピーパスに付随する、より詳細な概念と戦略の**説明**を意図したものです。

:::note
以下のセクションは、抽象度が高くなる順に列挙されています。
:::

### 式言語

[LangChain式言語(LCEL)](/docs/expression_language/)は、ほとんどのLangChainコンポーネントが組み合わさる基本的な方法であり、このセクションでは開発者がLangChainのプリミティブを効果的に使ってビルドする方法を教えることを目的としています。

このセクションには、LCEL プリミティブをストリーミングや抽象的なタスクに使う方法を教える**チュートリアル**、特定の動作を**説明**する内容、Runnableインターフェースの使い方に関する**リファレンス**が含まれるべきです。

### コンポーネント

[コンポーネントセクション](/docs/modules)は、LCELよりも抽象度の高い概念をカバーしています。
`BaseChatModel`や`BaseRetriever`などの抽象基底クラスや、`ChatPromptTemplate`や`RecursiveCharacterTextSplitter`などのこれらの基底クラスの具体的な実装、カスタマイズガイドなどがここに含まれます。

このセクションには、主に概念的な**チュートリアル**、**リファレンス**、カバーするコンポーネントの**説明**が含まれるべきです。

:::note
一般的なルールとしては、`Expression Language`と`Components`セクション(ただし`Composition`セクションを除く)でカバーされるすべてのものは、`langchain_core`に存在するコンポーネントのみを対象とすべきです。
:::

### インテグレーション

[インテグレーション](/docs/integrations/platforms/)は、コンポーネントの特定の実装です。これらには多くの場合、サードパーティのAPIやサービスが関係しています。
そうである場合、一般的なルールとしては、これらはサードパーティのパートナーによって管理されています。

このセクションには、主に**説明**と**リファレンス**が含まれるべきですが、実際のコンテンツはその他のセクションよりも柔軟で、サードパーティプロバイダーの裁量に任されています。

:::note
`Integrations`でカバーされる概念は、一般的に`langchain_community`または特定のパートナーパッケージに存在するべきです。
:::

### ガイドとエコシステム

[ガイド](/docs/guides)と[エコシステム](/docs/langsmith/)セクションには、上記のセクションよりも高レベルの問題に取り組むガイドが含まれるべきです。
これには、プロダクション化や開発ワークフローに関する考慮事項などが含まれます。

これらには主に**ハウツーガイド**、**説明**、**チュートリアル**が含まれるべきです。

### APIリファレンス

LangChainのAPIリファレンス。**リファレンス**として機能し(名称が示すとおり)、一部**説明**重視のコンテンツも含むべきです。

## サンプル開発者ジャーニー

私たちは新しい開発者がLangChainを使えるようにドキュメントを設定しました。意図されたパスを見ていきましょう:

- 開発者が https://python.langchain.com にアクセスし、イントロダクションと図を読みます。
- 単に好奇心があるだけの場合は、[クイックスタート](/docs/get_started/quickstart)に引き付けられ、LangChainの概要を把握できます。
- 特定のタスクを達成したい場合は、ユースケースセクションに引き付けられます。ユースケースは、LangChainが提供できる価値を具体的に示し、フレームワークの良い入り口となるはずです。
- その後、Expression Languageセクションを通してLangChainの基本を学びます。
- 次に、LangChainのさまざまなコンポーネントと統合について学びます。
- 最後に、ガイドを通して追加の知識を得ることができます。

これは理想的なものですが、各セクションは必然的に他のセクションに記載されている上位または下位のコンセプトを参照することになります。

## ガイドライン

ドキュメントの作成と整理を考える際のその他のガイドラインは以下の通りです。

### 他のセクションへのリンク

ドキュメントのセクションは真空状態にはないため、開発者が不慣れなトピックについてさらに学べるよう、できるだけ他のセクションにリンクすることが重要です。

これには、API リファレンスと概念セクションへのリンクが含まれます。

### 簡潔さ

一般的に、少ないほうが良いアプローチを取ります。既存のセクションに概念の良い説明がある場合は、その概念に新しい側面がない限り、再度説明するのではなく、そのセクションにリンクすべきです。

簡潔にする、コードサンプルも含めて。

### 一般的なスタイル

- 可能な限り能動態と現在形を使用する。
- 概念と使用法を説明するためにサンプルとコードスニペットを使用する。
- コンテンツを階層的に整理するために適切なヘッダーレベル (`#`, `##`, `###` など) を使用する。
- 情報を簡単に消化できるようにするためにブレットポイントと番号付きリストを使用する。
- 参照セクションでは特にテーブルと図を頻繁に使用して、情報を視覚的に表示する。
- 読者がコンテンツを簡単に移動できるよう、長いドキュメントページにはテーブルオブコンテンツを含めるが、短いページには含めない。
