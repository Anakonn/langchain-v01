# 기술적인 로지스틱스

LangChain 문서는 두 가지 구성 요소로 구성되어 있습니다:

1. 메인 문서: [python.langchain.com](https://python.langchain.com/)에 호스팅된 이 포괄적인 리소스는 주요 사용자 대상 문서입니다.
   튜토리얼, 사용 사례, 통합 등 다양한 주제를 다루며 LangChain으로 구축하는 방법에 대한 광범위한 지침을 제공합니다.
   이 문서의 콘텐츠는 모노레포의 `/docs` 디렉터리에 있습니다.
2. 코드 내 문서: 코드베이스 자체에 대한 문서로, 외부에서 볼 수 있는 [API 참조](https://api.python.langchain.com/en/latest/langchain_api_reference.html)를 생성하는 데 사용됩니다.
   API 참조의 콘텐츠는 코드베이스의 독스트링을 스캔하여 자동 생성됩니다. 따라서 개발자들은 코드를 잘 문서화해야 합니다.

메인 문서는 [Quarto](https://quarto.org) 및 [Docusaurus 2](https://docusaurus.io/)를 사용하여 작성됩니다.

`API 참조`는 주로 [sphinx](https://www.sphinx-doc.org/en/master/)에 의해 코드에서 자동 생성되며 [Read the Docs](https://readthedocs.org/)에 호스팅됩니다.

오타 수정, 새로운 튜토리얼 또는 예제 추가 등 모든 문서 기여를 환영합니다. 메인 문서든 API 참조든 상관없습니다.

린팅과 마찬가지로 문서 작성이 번거로울 수 있다는 것을 인식하고 있습니다. 문서를 작성하고 싶지 않다면 프로젝트 메인테이너에게 연락하세요. 도움을 드릴 수 있습니다. 좋은 코드 기여에 대한 방해물이 되지 않기를 바랍니다.

## 📜 메인 문서

메인 문서의 콘텐츠는 모노레포의 `/docs` 디렉터리에 있습니다.

문서는 ipython 노트북(`.ipynb` 파일)과 마크다운(`.mdx` 파일)을 조합하여 작성됩니다. 노트북은 [Quarto](https://quarto.org)를 사용하여 마크다운으로 변환된 후 [Docusaurus 2](https://docusaurus.io/)를 사용하여 빌드됩니다.

메인 문서에 기여하는 것을 환영합니다! 🥰

문서를 수정한 후:

1. 문서가 잘 포맷되고 오류가 없는지 확인하기 위해 린팅 및 포맷팅 명령을 실행하세요(아래 참조).
2. 선택적으로 로컬에서 문서를 빌드하여 변경 사항이 잘 보이는지 확인하세요.
3. 변경 사항으로 풀 리퀘스트를 만드세요.
4. 풀 리퀘스트 `대화` 페이지에서 `View deployment` 또는 `Visit Preview` 버튼을 클릭하여 변경 사항이 원하는 대로인지 미리 보고 확인할 수 있습니다. 이는 문서 변경 사항의 미리보기를 제공합니다.

## ⚒️ 로컬에서 문서 린팅 및 빌드

문서를 작성한 후, 문서가 잘 보이고 오류가 없는지 로컬에서 린팅 및 빌드를 원할 수 있습니다.

로컬에서 빌드할 수 없는 경우에도 괜찮습니다. 풀 리퀘스트 페이지에서 문서의 미리보기를 볼 수 있습니다.

### 종속성 설치

- [Quarto](https://quarto.org) - Jupyter 노트북(`.ipynb` 파일)을 Docusaurus에서 제공하기 위한 mdx 파일로 변환하는 패키지. [다운로드 링크](https://quarto.org/docs/download/).

**모노레포 루트**에서 다음 명령을 실행하여 종속성을 설치합니다:

```bash
poetry install --with lint,docs --no-root
```

### 빌드

문서를 빌드하는 코드는 모노레포의 `/docs` 디렉터리에 있습니다.

다음 명령에서 `api_` 접두사는 API 참조를 위한 작업을 나타냅니다.

문서를 빌드하기 전에 빌드 디렉터리를 정리하는 것이 좋습니다:

```bash
make docs_clean
make api_docs_clean
```

다음으로, 아래에 설명된 대로 문서를 빌드할 수 있습니다:

```bash
make docs_build
make api_docs_build
```

마지막으로, 모든 링크가 유효한지 확인하려면 링크 체커를 실행하세요:

```bash
make docs_linkcheck
make api_docs_linkcheck
```

### 린팅 및 포맷팅

메인 문서는 **모노레포 루트**에서 린팅됩니다. 메인 문서를 린팅하려면 다음 명령을 실행하세요:

```bash
make lint
```

포맷팅 관련 오류가 있는 경우 다음 명령으로 자동으로 수정할 수 있습니다:

```bash
make format
```

## ⌨️ 코드 내 문서

코드 내 문서는 주로 [sphinx](https://www.sphinx-doc.org/en/master/)에 의해 코드에서 자동 생성되며 [Read the Docs](https://readthedocs.org/)에 호스팅됩니다.

API 참조가 유용하려면 코드베이스가 잘 문서화되어야 합니다. 이는 모든 함수, 클래스 및 메서드에 해당 기능이 무엇을 하는지, 인수가 무엇인지, 반환 값이 무엇인지 설명하는 독스트링이 있어야 함을 의미합니다. 이는 일반적으로 좋은 관행이지만, LangChain에서는 특히 중요합니다. 왜냐하면 API 참조가 개발자가 코드베이스를 이해하는 주요 리소스이기 때문입니다.

우리는 일반적으로 [Google Python Style Guide](https://google.github.io/styleguide/pyguide.html#38-comments-and-docstrings)를 따릅니다.

다음은 잘 문서화된 함수의 예입니다:

```python
def my_function(arg1: int, arg2: str) -> float:
    """이 함수의 간단한 설명. (단일 문장이어야 합니다.)

    이 함수의 더 긴 설명입니다. 함수가 무엇을 하는지, 인수가 무엇인지,
    반환 값이 무엇인지 설명해야 합니다. 88자에서 줄 바꿈을 해야 합니다.

    예제:
        함수 사용 방법에 대한 예제 섹션입니다.

        .. code-block:: python

            my_function(1, "hello")

    Args:
        arg1: arg1에 대한 설명입니다. 함수 서명에 이미 지정되어 있기 때문에 유형을 지정할 필요가 없습니다.
        arg2: arg2에 대한 설명입니다.

    Returns:
        반환 값에 대한 설명입니다.
    """
    return 3.14
```

### 린팅 및 포맷팅

코드 내 문서는 문서화되는 패키지 디렉터리에서 린팅됩니다.

예를 들어, `langchain-community` 패키지에서 작업 중이라면 작업 디렉터리를 `langchain-community` 디렉터리로 변경합니다:

```bash
cd [root]/libs/langchain-community
```

패키지에 대한 가상 환경을 아직 설정하지 않았다면 설정하세요.

패키지에 대한 종속성을 설치합니다.

```bash
poetry install --with lint
```

그런 다음 다음 명령을 실행하여 코드 내 문서를 린팅하고 포맷팅할 수 있습니다:

```bash
make format
make lint
```

## 문서 변경 사항 확인

문서 변경 사항을 리포지토리에 푸시한 후, 풀 리퀘스트 `대화` 페이지에서 `View deployment` 또는 `Visit Preview` 버튼을 클릭하여
변경 사항이 원하는 대로인지 미리 보고 확인할 수 있습니다.
이 미리보기는 [Vercel](https://vercel.com/docs/getting-started-with-vercel)에 의해 생성됩니다.
